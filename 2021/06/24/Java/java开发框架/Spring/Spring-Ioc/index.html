Spring的IOC

2021-06-24

IOC

  控制反转 IOC

  Spring Ioc容器控制bean的生命周期。即bean的创建，销毁等。

  依赖注入 DI

  通过接口，构造方法，注解等方法，由Ioc容器动态的将某个依赖，注入到组件中。相当于bean之间的关系由容器完成。

bean的生命周期


  在bean执行构造方法实例化后，通过依赖注入填充属性，而此阶段可能发生循环依赖。如果bean实现了Aware接口，则按图中顺序执行实现的方法。
  BeanPostProcessor就是后置处理器，AOP就是在初始化后置阶段实现。BeanPostProcessor是针对所有的bean的，每个bean在生命周期中，都会执行所有的BeanPostProcessor中方法，包括实现接口自定义的BeanPostProcessor。可以在方法中添加条件判断，让自定义的BeanPostProcessor跳过不符合条件的Bean。
  bean的初始化阶段，就是在使用bean之前，执行初始化方法中的内容，来扩展功能，如读取mysql中数据。

三级缓存解决循环依赖

  提前曝光

  解决循环依赖，核心即为bean的提前曝光。只执行构造方法，完成对象创建，没有执行注入依赖及初始化。

  循环依赖的发现机制


  当生成A的bean时，会通过 isSingletonCurrentlyInCreation() 方法先判断该beanName对应的bean是否处于正在创建，如果没有就将该beanName添加到一个set集合中，表示该beanName的bean正在创建，当bean生成流程全部执行完时再移除集合。
  通过 createBeanInstance() 方法实例化bean，然后通过 populateBean() 方法进行填充属性，此时无法直接给字段b填充B的bean，因此开始实例化B的bean并填充属性，且在实例化该bean时，将该bean的beanName添加到**”正在创建”set集合**中。
  此时填充属性发现，B依赖的A存在于set集合中，表示A正在创建中。因此发现发生了循环依赖。

  一级缓存 singletonObjects


  假设A，B，C之间的关系如图所示。创建B的bean时，在填充属性时会创建一个A的bean。当创建C的bean时，在填充属性时也需要A的bean。此时如果再创建一个A的bean，就破坏了beanName和bean之间的单例关系。因此需要一个容器保存创建完成的bean。
  使用ConcurrentHashMap保存完整的bean。

  二级缓存 earlySingletonObjects


  假设A，B，C之间的关系如图所示。在创建A的bean时，在填充属性时会创建B的bean。而B的bean时需要用到A的bean，而此时A的bean还没有完成。为了进行下去，就需要用到二级缓存，来保存实例化，但未注入依赖的bean，也就是不完整的bean。
  使用HashMap保存不完整的bean。只有一级缓存是用concurrentHashMap，其他使用的为HashMap，但是会通过synchronized同步锁来实现线程安全。

  三级缓存 singletonFactories

  解决循环依赖的核心是提前曝光，也就是在bean构造后，就要有一个容器来保存，以供后面可能发生的循环依赖来使用，也就是singletonFactories。在构造阶段，Spring不知道是否会发生循环依赖。如果bean涉及到AOP，也就是需要代理对象。
  正常流程中，生成代理对象bean，是在完成初始化等操作后，添加到singletonObjects之前执行。Spring采取发生依赖循环时将AOP生成的代理对象bean提前。因此三级缓存提前曝光的是SingletonFactory对象。
  遇到循环依赖时，如果需要提前aop代理bean，singletonFactory.**getObject()**创建代理bean，并保存到earlySingletonObjects中。如果不涉及到AOP，SingletonFactory本来就保存了原始bean，getObject()方法返回原始bean，保存到earlySingletonObjects中。

  三级缓存 和 二级缓存

  三级缓存中，提前曝光的是singletonFactory对象。发生循环依赖时通过该对象可以得到，bean或者代理bean，再加入到二级缓存中。
  如果没有发生循环依赖，那么bean的生命周期过程不受影响。只有发生循环依赖且涉及到AOP生成代理对象，才会影响本来的生命周期，需要将AOP提前。
  如果不采用三级缓存，取消掉singletonFactories。那么需要将earlySingletonObjects提前，在bean构造后就直接保存到earlySingletonObjects中，提前曝光的是未完成的bean。因为在构造阶段不知道是否会发生循环依赖的。在只有二级缓存中，如果涉及到AOP，那么必须在bean构造后就生成代理bean，并将代理bean保存到earlySingletonObjects中。就算没有发生循环依赖，那也已经AOP提前了，破化了原来的生命周期流程。
